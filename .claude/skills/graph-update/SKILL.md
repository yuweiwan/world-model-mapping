# 图谱更新 Skill

## 核心理念

你是世界模型知识图谱的**策展人**，不是搬运工。每添加一个节点，都应满足：

- **质量优先于数量**：只添加与图谱中已有实体有明确强关联的节点。弱关联（"同为XX"、"都参加过YY会议"）不建边。
- **一手信息优先**：公司信息看官网/about页，学术背景查论文/Google Scholar/个人主页，切忌采用 CSDN、阿里云/腾讯云/华为云社区等内容农场。
- **关联必须具体**：每条边都要有明确的 `label` 说明关系性质，不能笼统地写"关联"。
- **格式零偏差**：节点 ID、字段名、relation_type 必须符合 `references/graph-schema.md` 规范。
- **增量更新**：只动需要改的部分，不要碰无关节点和边。

## 前置条件

更新图谱前，确认：
1. `graph-data.js` 格式为 `window.GRAPH_DATA = { "nodes": [...], "links": [...] };`
2. 已读取 `references/graph-schema.md` 了解完整格式规范
3. 已读取 `references/node-naming.md` 了解命名规则
4. 备好联网搜索能力（WebSearch / WebFetch / CDP，按需选用）

## 工作流程

### 输入类型判断

根据用户输入自动判断类型，选择对应入口：

| 类型 | 典型输入 | 入口 |
|------|---------|------|
| **A. 文章提取** | 一个 URL（微信文章、论文链接、新闻报道） | → Step A |
| **B. 人物调研** | 一个人名 + "加入图谱" / "查一下背景" | → Step B |
| **C. 主题补充** | 一个关键词 + "搜索补充" / "还有什么遗漏" | → Step C |

---

### Step A：从文章中提取实体

**A1. 获取文章内容**
- 普通网页 → `WebFetch`（传入 URL + 提取 prompt）
- 微信公众号等反爬站点 → CDP 浏览器（`/new` → `/eval` 提取 `#js_content` 的 innerText）
- 内容过长时分段处理，先通读全文再提取

**A2. 逐段提取实体**
按三类分别提取：
- **人物**：姓名 → 所属机构 → 研究方向 → 师承关系 → 重要成果
- **公司/机构**：全称 → 所属路线 → 核心产品/业务
- **产品/论文**：全称 → 年份 → 作者 → 所属路线 → 核心贡献

**A3. 去重检查**
对每个提取到的实体，在 `graph-data.js` 中搜索是否已存在：
- 按 `name` 字段模糊匹配（中英文名、别名）
- 搜索不到 → 标记为"需新建"
- 搜索到 → 标记为"已存在"，检查是否需要补充信息（如新的任职、新的成果）

**A4. 关联映射**
对每个"需新建"的实体，找出与已有节点的关联：
- 文中明确提到的关系（导师-学生、同事、创办、任职）→ 直接建边
- 文中暗示的关系 → 用 WebSearch 确认真实性后再建边
- 无法确认的弱关联 → 不建边

**A5. 生成并保存** → 跳到 Step 通用-保存

---

### Step B：调研单个人物并加入图谱

**B1. 消歧与定位**
- 先确认是哪个"XX"（同名人物很多，需结合机构/领域消歧）
- 用 WebSearch 搜索 "{姓名} {机构/领域}" 确认身份

**B2. 深度调研**
至少查清以下维度（用 WebSearch + WebFetch 交叉验证）：

| 维度 | 来源 | 示例问题 |
|------|------|---------|
| 教育背景 | 个人主页、Google Scholar | 博士导师是谁？毕业院校？ |
| 职业经历 | LinkedIn、公司官网/about | 在哪些公司/机构任职过？ |
| 研究方向 | Google Scholar、论文列表 | 核心方向是什么？发了哪些重要论文？ |
| 师承关系 | 学术谱系（Math Genealogy）、论文致谢 | 导师是谁？带过哪些学生？ |
| 合作网络 | 共同作者、项目合作 | 和谁合作密切？ |
| 创业/产业 | 新闻报道、Crunchbase | 是否创办过公司？ |

**B3. 发现与已有节点的关联**
- 在调研结果中查找：这个人是否与图谱中已有的 569 个节点有直接关联？
- 常见的关联类型：师徒、同事、合作者、创办者、任职者
- 如果有：生成对应的边

**B4. 生成并保存** → 跳到 Step 通用-保存

---

### Step C：主题/关键词补充

**C1. 搜索主题**
- 用 WebSearch 搜索 "{主题} world model / 世界模型"
- 收集该主题下的关键人物、公司、产品/论文

**C2. 对比已有图谱**
- 检查搜索结果中的实体哪些已在图谱中
- 列出遗漏的实体

**C3. 逐一调研遗漏实体**
- 对每个遗漏实体，按 Step B 的流程调研

**C4. 生成并保存** → 跳到 Step 通用-保存

---

### Step 通用-保存：生成节点和边 → 校验 → 提交

**1. 生成节点 JSON**

每个新节点必须包含以下字段，参照 `references/graph-schema.md`：

```json
{
  "id": "<按 naming 规则生成>",
  "name": "<显示名称，中国人用 中文+英文，外国人用 英文>",
  "type": "person|company|product_paper",
  "description": "<100-200字简介，包含关键时间、机构、成果>",
  "degree": 0,
  "composite_weight": 0.0,
  "group_id": "<jepa|spatial|sim|active_inf|causal>",
  "group_name": "<对应中文路线名>",
  "tags": ["<路线名>", "<可选额外标签>"]
}
```

**2. 生成边 JSON**

```json
{
  "source": "<已有或新节点的 id>",
  "target": "<已有或新节点的 id>",
  "relation_type": "mentors|collaborates|founded|works_at|develops|evolves_to|related",
  "label": "<具体关系说明，如 'PhD advisor'、'联合创始人'、'共同发表'>",
  "weight": 2
}
```

**边的权重指南：**

| 权重 | 使用场景 |
|------|---------|
| 1 | 一般关联（间接、较弱） |
| 2 | 明确关系（同事、一般合作） |
| 3 | 重要关系（核心合作者、重要产品推动） |
| 4 | 强关系（直属导师、联合创始人、核心团队成员） |
| 5 | 极强关系（创始人-公司、开创性论文-作者） |

**3. 插入数据**

使用 Node.js 脚本操作 `graph-data.js`：
1. `JSON.parse` 解析 `window.GRAPH_DATA = {...}` 的数据部分
2. 将新节点追加到 `nodes` 数组末尾
3. 将新边追加到 `links` 数组末尾
4. 重新计算所有节点的 `degree`（统计每个节点在 links 中出现的次数）
5. 计算 `maxDegree`，对每个节点 `composite_weight = degree / maxDegree`
6. 以 2 空格缩进的 JSON 格式写回 `graph-data.js`，保留 `window.GRAPH_DATA = ` 前缀

**4. 校验清单**

保存前必须逐项确认：

- [ ] 所有新增 `source` / `target` 对应的节点 ID 在 nodes 数组中存在
- [ ] 所有 `relation_type` 是 7 种值之一（mentors / collaborates / founded / works_at / develops / evolves_to / related）
- [ ] 新节点 `id` 不与已有节点重复
- [ ] 新边的 `(source, target, relation_type)` 三元组不与已有边重复
- [ ] 所有节点的 `degree` 已从实际边数重新计算
- [ ] 所有节点的 `composite_weight` 已统一重算
- [ ] `graph-data.js` 是合法的 JSON（`JSON.parse` 不报错）
- [ ] `World_Models_TechMap.html` 中的实体数/关系数统计已更新匹配

**5. 提交**

每次更新独立 commit，message 格式：
```
feat: 从[来源]新增 X 个节点、Y 条边 — [简要说明]
```

例如：
```
feat: 从"世界模型五大学派"文章新增 12 个节点、18 条边 — 潘恒楷、Gaoyue Zhou、姜云凡等
```

然后 `git push origin main`。

## 7 种关系类型速查

| 类型 | 中文 | 方向 | 适用场景 |
|------|------|------|---------|
| `mentors` | 指导 | 导师 → 学生 | PhD advisor、博士后导师、学术引路人 |
| `collaborates` | 合作 | 人物 ↔ 人物/机构 | 共同发表、项目合作、联合创办 |
| `founded` | 创办 | 人物 → 公司/机构 | 创始人、联合创始人 |
| `works_at` | 任职 | 人物 → 公司/机构 | 教授、研究员、工程师、VP、CEO |
| `develops` | 开发/推动 | 人物 → 产品/概念 | 论文作者、产品负责人、核心推动者 |
| `evolves_to` | 演变 | 产品 → 后继产品 | 技术代际演进（DreamerV3 → Dreamer4） |
| `related` | 关联 | 概念 ↔ 概念 | 理论关联、技术同源（**慎用**：只用于有明确学术/技术渊源的关联） |

### 关系方向规则

- **有明确主动方的**：箭头从主动方指向被动方（导师→学生，创始人→公司，开发者→产品）
- **平等的**：`collaborates`、`related`，方向不重要，但要一致
- **一个人在一个公司任职**：`works_at` (人→公司)
- **一个人创办一个公司**：`founded` (人→公司)
- **一个人开发/推动一个产品/概念**：`develops` (人→产品)
- **不要反向建边**：不需要 "被指导"、"被创办" 等反向关系

## 5 条技术路线速查

| group_id | group_name | 核心主题 |
|----------|-----------|---------|
| `jepa` | 路线一：JEPA 联合嵌入预测架构 | LeCun 隐式表征、自监督学习、视频预测 |
| `spatial` | 路线二：空间智能（3D 世界模型） | 李飞飞 3D 重建、神经渲染、VGGT |
| `sim` | 路线三：学习仿真（生成视频 + RL + 基础设施） | NVIDIA Cosmos、具身智能、Sim-to-Real |
| `active_inf` | 路线四：主动推理（边缘路线） | Friston 自由能原理 |
| `causal` | 路线五：因果推断（Pearl 因果推理框架） | 因果推理、结构化因果模型 |

### 新节点路线分配原则

- 人物：按其核心研究方向分配（不确定时看其主要成果/论文属于哪条路线）
- 公司：按其核心产品/技术路线分配
- 产品/论文：按其技术路线分配
- 如果一个实体跨越多条路线，选择最核心的一条，在 tags 中追加其他路线标签

## 反模式

| ❌ 错误做法 | ✅ 正确做法 |
|------------|-----------|
| 看到文章提到一个人就直接加 | 先去重，确认不在图谱中才新建 |
| 中文名称当 ID（如 `王昊`） | 用英文命名规则（`Wang_Hao`） |
| 两个人"同在一个领域"就建 `collaborates` 边 | 需要具体合作证据（共同论文、项目等） |
| "同为XX公司成员"就建 `related` 边 | 弱的间接关联不建边，保持图谱清爽 |
| 描述太长（300+ 字） | 控制在 100-200 字，抓关键信息 |
| 忘记更新 HTML 统计数字 | 每次更新节点/边数后同步更新 HTML |
| 边没有 label | 每条边必须有具体的关系说明 |
| 用小数权重（0.5, 0.8 等） | 统一用整数权重（1-5） |
| 一个 commit 包含多个不相关的更新 | 一个 commit 对应一次更新操作 |
| 跳过校验直接提交 | 必须先跑校验清单 |
